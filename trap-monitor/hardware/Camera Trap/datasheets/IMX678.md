### Spectral Response — CONFIRMED from Production Datasheet (E21Y08A29 Page 22)

The spectral sensitivity chart (wafer-level) shows:
- **Peak sensitivity** at ~500nm (green) and ~800nm (NIR) — the characteristic STARVIS 2 dual-peak
- All three channels (R, G, B) converge above 750nm where the Bayer filter becomes transparent
- **At 850nm:** relative response ~0.85 (all channels) — excellent for 850nm IR illumination
- **At 940nm:** relative response ~0.45-0.50 (all channels) — usable for 940nm covert IR
- **At 1000nm:** relative response ~0.15-0.20 — dropping off but still detectable
- **Cutoff ~1050nm** — consistent with silicon bandgap

> This confirms that 940nm covert IR illumination will produce usable images. The STARVIS 2 uneven photodiode surface enhances NIR absorption — the 940nm response is approximately 2× better than comparable STARVIS 1 sensors at the same pixel size.

## Power-On Sequence

**Sequence CONFIRMED from production datasheet E21Y08A29 Pages 41-43:**

1. **DVDD (1.1V)** — enable first
2. **DOVDD (1.8V)** — enable second
3. **AVDD (3.3V)** — enable third
4. All rails must be stable within 200 ms of sequence start (unverified — commonly cited but may be from preliminary docs)
5. After all rails stable, release XCLR (drive HIGH)
6. Wait for sensor internal PLL lock before I2C access

**Power-off sequence:** Reverse order (assert XCLR LOW, then disable AVDD → DOVDD → DVDD).

**From Linux driver timing constants:**
- XCLR release delay: 500 ms (conservative — driver uses 500,000 µs)
- XCLR delay range: 1 ms
- Stream enable delay: 25 ms after mode registers written
- Stream delay range: 1 ms

### HDR Modes — from Sony Flyer Page 1

- **DOL HDR** (Digital Overlap): Two successive exposures (short + long), synthesized. Slight time lapse between shots — can cause motion artifacts.
- **Clear HDR**: Two simultaneous captures at different gain levels (low for bright regions, high for dark regions), synthesized. No motion artifacts — better for moving targets like animals.

> **For this camera trap:** Clear HDR is the preferred mode if HDR is implemented (Phase 2). DOL HDR would cause motion blur on fast-moving animals. However, HDR halves frame rate — at 72fps base, Clear HDR gives 36fps. Not planned for Phase 1.

### STARVIS 2 Technology — from Sony/Macnica Technology Guide

- Back-illuminated pixel with deeper photodiodes + uneven surface textures
- NIR sensitivity enhanced by refracting incident light deeper into silicon
- Sensitivity ≥2000 mV/µm² (706 cd/m², F5.6, 1s accumulation)
- Dynamic range >8dB wider than STARVIS 1 at same pixel size (12-bit single exposure)
- NIR improvement comes from photodiode structure, not filter — works across entire NIR band to silicon cutoff (~1050nm)

## Application Notes for This Design

### Power Rail Implementation
- **AVDD (3.3V):** Fed from AP63300 3.3V rail (U11) via ferrite bead + LC filter for analog noise isolation
- **DVDD (1.1V):** Dedicated TPS62088 (U10b). Cannot share with STM32N6 VDDCORE (0.81V/0.89V)
- **DOVDD (1.8V):** TPS7A02 LDO (U12), shared with PSRAM VDD/VDDQ and STM32N6 VDDIO_XSPI1. Low noise critical for image quality.

### Level Shifting
- I2C (SDA/SCL): BSS138 discrete level shifters, 3.3V MCU side ↔ 1.8V sensor side
- XCLR: Open-drain GPIO (PC8) with 10 kΩ pull-up to 1.8V
- PWDN: Open-drain GPIO (PD2) with 10 kΩ pull-up to 1.8V

### Clock
- Dedicated 24 MHz MEMS oscillator (SiT8008, Y2) wired directly to INCK
- Oscillator powered from 1.8V rail — power-gated with sensor
- No MCU clock output used (MCO1 has no usable pin on this package)

### MIPI CSI-2
- 2-lane mode used in this design (STM32N6 CSI-2 interface is 2-lane)
- Dedicated D-PHY pads on STM32N6 (not GPIO-muxed)
- 100 Ω differential impedance routing required
- Target link frequency: 720 MHz (1440 Mbps/lane) for 4K @ ~33 fps

## Known Issues / Errata

1. **No chip ID register.** The IMX678 lacks a dedicated chip ID register (unlike many other sensors). The Linux driver reads the black level register (0x30DC) to verify I2C bus connectivity. This is adequate but not a true identification mechanism.

2. **NDA barrier.** Full register documentation, timing diagrams, and electrical limits require NDA. The Linux driver provides a practical register map, but bit-field details for some registers are incomplete.

3. **XCLR timing is conservative in the Linux driver.** The driver uses a 500 ms delay after XCLR release. Sony's actual requirement may be shorter (likely 10-50 ms for PLL lock), but without NDA datasheet confirmation, the conservative delay is prudent.

4. **ClearHDR mode halves frame rate.** If HDR is enabled, expect ~50% frame rate reduction. For this camera trap application, HDR is not planned for Phase 1.

5. **PWDN pin polarity.** Active HIGH = power-down/standby. This is the opposite convention from some other Sony sensors (e.g., IMX219 where PWDN is active LOW). The design correctly uses a pull-up to 1.8V (standby on boot) and drives LOW to activate.
