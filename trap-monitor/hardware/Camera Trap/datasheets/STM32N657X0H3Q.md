# STM32N657X0H3Q — Cortex-M55 MCU with Neural-ART NPU, 4.2MB SRAM

High-performance microcontroller with hardware AI accelerator, ISP, JPEG/H.264 codec, and MIPI CSI-2 camera interface. Used as the main MCU in the Between 23 AI Camera Trap.

### VDDCORE Details

- **VOS nominal (0.81V):** CPU runs up to 600 MHz. Lower power.
- **VOS overdrive (0.89V):** CPU runs at full 800 MHz, NPU at full speed.
- **Switching:** GPIO PF4 controls TPS62088 feedback divider on DK board.
- **Feeding >0.89V to VDDCORE is out of spec.** Our V1.1 error (1.1V to VDDCORE) would have exceeded abs max.

### HSLV OTP Fuses (CRITICAL — Irreversible)

When VDDIO2 or VDDIO3 is operated at 1.8V, the corresponding HSLV (High-Speed Low-Voltage) OTP fuse **must** be programmed in OTP124:

| OTP124 Bit | Domain | Purpose |
|------------|--------|---------|
| HSLV_VDDIO2 | VDDIO2 (XSPI P1) | Enable 1.8V high-speed I/O for HexaSPI |
| HSLV_VDDIO3 | VDDIO3 (XSPI P2) | Enable 1.8V high-speed I/O for OctoSPI |
| HSLV_VDDIO4 | VDDIO4 (SDMMC1) | Enable 1.8V high-speed I/O if needed |
| HSLV_VDDIO5 | VDDIO5 (SDMMC2) | Enable 1.8V high-speed I/O if needed |
| HSLV_VDD | VDD | Enable 1.8V for main VDD domain |

**WARNING:** Setting HSLV_VDDIOx while VDDIOx is at 3.3V **permanently damages the device**. OTP fuses are one-time-programmable and cannot be reversed. The corresponding PWR_SVMCR register bits must match the OTP fuse state.

**For our design:** VDDIO2 (XSPI P1 = PSRAM) and VDDIO3 (XSPI P2 = NOR flash) run at 1.8V. HSLV fuses for both must be programmed during provisioning. VDDIO4/5 (SDMMC) at 3.3V — do NOT fuse.

### Low-Power Modes

**IMPORTANT: The exact current figures below are from our power budget estimates and ST community posts. Verify against datasheet electrical characteristics tables.**

| Mode | Conditions | Current | Confidence |
|------|-----------|---------|------------|
| Standby (backup RAM ON, RTC/LSE ON, VDD=1.8V) | **48.4 µA typ, 69.0 µA max @ 25°C, 339 µA max @ 85°C** | **CONFIRMED** DS14643 Table 44 Page 157 |

> **CRITICAL DESIGN DECISION:** The power budget assumed "STM32N6 deep sleep ~8µA". The actual standby current is 48µA typ. VBAT mode (VDD off) is 37-76µA depending on backup RAM.
>
> **However** — if the N6's VBAT pin is left unpowered during sleep (U0 handles all timekeeping via its own RTC), the N6 draws **0µA**. The question is: does the N6 need anything from its VBAT domain during sleep? If not, disconnect VBAT from the always-on rail and gate it with the N6 power enable. This recovers the full 37-76µA.
>
> **Options:**
> 1. **VBAT unpowered (0µA):** N6 loses RTC and backup registers on every sleep. U0 provides all timekeeping. N6 cold-boots on every wake. Best for power.
> 2. **VBAT powered, no backup RAM (37µA max):** N6 keeps RTC but loses SRAM. Still cold-boots but retains timestamp.
> 3. **VBAT powered + backup RAM (76µA max):** N6 keeps 8KB backup SRAM. Could store wake context. Expensive.
>
> **Recommendation:** Option 1 — VBAT unpowered. The U0 already has its own RTC for wake scheduling. The N6 doesn't need to keep time during sleep. Gate VBAT with the N6 power enable signal.

| VBAT mode (backup RAM ON, RTC/LSE ON, VBAT=3.3V) | **75.8 µA max @ 25°C** | **CONFIRMED** Table 45 |

### Power-On Sequence

**CRITICAL — must be followed or device may not boot:**

1. **VDDCORE** must be stable before or simultaneously with VDD
2. **VDD** (1.71-3.6V) powers up
3. **VDDIOx** domains power up (can be sequenced per bank)
4. **VBAT** can be connected before VDD (backup domain)

**From AN6000:** The discrete power supply must ensure VDDCORE reaches target voltage within specification before VDD rises above POR threshold. The DK board uses TPS62088 for VDDCORE with enable tied to the 3.3V rail through a delay circuit.

**For our cascaded design:**
1. Battery -> AP63300 U11 -> 3.3V (always on)
2. 3.3V -> TPS62088 U10a -> 0.81V/0.89V VDDCORE (follows 3.3V with ~1ms delay)
3. 3.3V -> TPS7A02 U12 -> 1.8V (DOVDD, PSRAM, VDDIO_XSPI)
4. POR on STM32N6 triggers when VDD crosses threshold

### Secure Boot Requirement

The STM32N6 has **no internal flash**. It boots from external XSPI memory via a ROM bootloader. The boot sequence is:

1. ROM boot code runs from internal ROM
2. Loads FSBL (First Stage Boot Loader) into AXISRAM2
3. FSBL loads application from external flash into AXISRAM1
4. Application runs from SRAM (XIP from external flash also possible but slower)

**Secure boot is mandatory in production.** The ROM bootloader verifies FSBL signature using keys stored in OTP. For development, unsigned boot is available via STM32CubeProgrammer.

## Application Notes for This Design

### Power Architecture (from AN6000 + our V1.2 review)

Our design uses a cascaded discrete power supply:
- **Stage 1 (from 6S2P battery 6.0-8.4V):** Three AP63300 bucks (3.3V always-on, 3.8V modem, 5.0V LED)
- **Stage 2 (from 3.3V):** Two TPS62088 (VDDCORE 0.81/0.89V, IMX678 DVDD 1.1V) + TPS7A02 (1.8V)

**Key gotchas:**
- TPS62088 max Vin = 5.5V. **Must be fed from 3.3V rail, never directly from battery.**
- VDDCORE at 0.81V for nominal (600MHz), 0.89V for overdrive (800MHz). **NOT 1.1V or 1.2V.**
- PF4 GPIO switches between voltage levels at runtime (before enabling NPU full speed)
- AP63300 Iq = 22uA — dominates sleep budget

### HSLV Configuration

For our 1.8V XSPI connections (PSRAM on XSPI1, NOR flash on XSPI2):
1. Program HSLV_VDDIO2 and HSLV_VDDIO3 in OTP124 during initial provisioning
2. Set matching bits in PWR_SVMCR3 register in FSBL
3. **Never apply 3.3V to VDDIO2/VDDIO3 after fusing** — permanent damage

### Memory Strategy

- Models (~4.0MB total) cannot fit simultaneously in 4.2MB SRAM with application code
- Load MegaDetector v5 (~2.5MB) first, run detection on 3 frames
- If animal detected, swap to AWC species classifier (~1.5MB), run classification
- Sequential loading adds ~50-100ms per swap from NOR flash via XSPI2
