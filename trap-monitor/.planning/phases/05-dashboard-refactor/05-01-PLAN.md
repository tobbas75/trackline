**A. Fix UnitGrid.test.tsx mock type error**

The `DefaultProps` interface uses `ReturnType<typeof vi.fn>` for callback props like `onSelect`, `onStatusFilterChange`, etc. This produces a type `Mock<Procedure | Constructable>` which does not match the expected function signatures (e.g., `(unitId: string) => void`).

Fix the `createDefaultProps` function by casting the mock functions to the correct types. Change the `DefaultProps` interface to use the actual function signatures from `UnitGridProps`, and cast `vi.fn()` accordingly:

```typescript
import type { UnitGridProps } from './UnitGrid';

// Remove the custom DefaultProps interface entirely. Use UnitGridProps directly:
function createDefaultProps(overrides?: Partial<UnitGridProps>): UnitGridProps {
  const units = [
    createUnit({ id: 'UNIT_001', name: 'Trap Alpha' }),
    createUnit({ id: 'UNIT_002', name: 'Trap Bravo' }),
  ];
  return {
    units,
    events: [],
    filteredUnits: units,
    selected: null,
    onSelect: vi.fn(),
    statusFilter: 'all',
    onStatusFilterChange: vi.fn(),
    searchQuery: '',
    onSearchQueryChange: vi.fn(),

    onSortKeyChange: vi.fn(),

}
```

For this to work, `UnitGridProps` must be exported from `UnitGrid.tsx`. Add the `export` keyword to the `UnitGridProps` interface declaration in `UnitGrid.tsx`:

Change: `interface UnitGridProps {`

The UI review flagged that ARM/DISARM commands fire immediately on click with no confirmation, which has real-world consequences for wildlife monitoring. Add a browser-native `window.confirm()` dialog before sending ARM or DISARM commands.

**In `frontend/src/app/dashboard/page.tsx`:**

In the command buttons section (currently line ~153-155), the code maps over `["STATUS", "GPS", unit.armed ? "DISARM" : "ARM"]` and calls `sendCommand(unit.id, cmd)` on click.

Change the `onClick` handler to check if the command is ARM or DISARM and show a confirmation dialog:

```tsx
{["STATUS", "GPS", unit.armed ? "DISARM" : "ARM"].map((cmd) => (
  <button
    key={cmd}
    onClick={() => {
      if (cmd === "ARM" || cmd === "DISARM") {
        const confirmed = window.confirm(
          `Are you sure you want to ${cmd.toLowerCase()} unit "${unit.name || unit.id}"?`
        );
        if (!confirmed) return;
      }
      sendCommand(unit.id, cmd);
    }}
    aria-label={`Send ${cmd} command to ${unit.name || unit.id}`}
    className="rounded-md border border-(--tm-border) bg-(--tm-panel) px-2 py-1.5 text-xs font-semibold text-(--tm-text) transition-colors hover:bg-(--tm-panel-soft)"
  >
    {cmd}
  </button>
))}
```

Use `window.confirm()` rather than a custom modal component because:
- This is a hardening milestone, not a feature milestone — no new UI components
- `window.confirm()` is blocking, accessible, keyboard-navigable, and works on mobile
- The confirmation only applies to ARM/DISARM, not STATUS/GPS (which are read-only queries)

Keep the file under 200 lines. The added lines for the confirm logic should be minimal (3-4 lines inside the onClick). If the file goes slightly over 200 lines, compress whitespace in the onClick handler to keep it on fewer lines.

Verify the final line count with `wc -l`.
