  <action>
  **STATE_CMD_LISTEN changes (per FW-03 -- grace period):**

  Replace the current STATE_CMD_LISTEN block with deadline-based logic. The key change: instead of comparing `millis() - start < fixed_window`, track a movable `deadline` variable. When a command is received and fewer than `CMD_LISTEN_GRACE_S` seconds remain, extend the deadline forward.

  New implementation:

  ```cpp
  case STATE_CMD_LISTEN:
  {
    unsigned long startTime = millis();
    unsigned long deadline = startTime + (CMD_LISTEN_WINDOW_S * 1000UL);
    unsigned long maxDuration = CMD_LISTEN_WINDOW_S * 3UL * 1000UL; // Safety cap: 3x base window
    Serial.printf("[CMD] Listening for %ds (grace: %ds, max: %lus)\n",
                  CMD_LISTEN_WINDOW_S, CMD_LISTEN_GRACE_S, maxDuration / 1000UL);

    while (millis() < deadline)
    {
      String inbound = modemCheckIncomingSMS();
      if (inbound.length() > 0)
      {
        Serial.printf("[CMD] Inbound: %s\n", inbound.c_str());
        String response = commandProcess(inbound, cfg, rtc);
        if (response.length() > 0)
        {
          modemSendSMS(cfg.smsNumber, response);
        }

        // Grace period: extend if command arrived near end of window (FW-03)
        unsigned long remaining = deadline - millis();
        unsigned long graceMs = CMD_LISTEN_GRACE_S * 1000UL;
        unsigned long elapsed = millis() - startTime;
        if (remaining < graceMs && elapsed < maxDuration)
        {
          deadline = millis() + graceMs;
          Serial.printf("[CMD] Grace period -- extended listen (%lus elapsed)\n", elapsed / 1000UL);
        }
      }
      delay(1000);
    }
    Serial.println("[CMD] Listen window closed");
    state = STATE_MODEM_OFF;
    break;
  }
  ```

  Key behaviors to ensure:
  - `deadline` starts at `millis() + CMD_LISTEN_WINDOW_S * 1000UL` (60 seconds by default)
  - Grace extension triggers ONLY when: (a) a command was just received, AND (b) `remaining < graceMs`, AND (c) total elapsed time < maxDuration (3x safety cap)
  - Each qualifying command re-extends the deadline to `millis() + graceMs` (10 seconds)
  - The safety cap (`maxDuration = CMD_LISTEN_WINDOW_S * 3UL * 1000UL`) prevents infinite extension if commands keep arriving
  - When no command arrives, the loop exits at the original deadline -- no change in behavior
  - A command received at second 50 (10s remaining, within grace zone) extends deadline to second 60 from that moment
  - A command received at second 30 (30s remaining, outside grace zone) does NOT extend -- remaining > graceMs
  - Serial logging indicates when grace extension occurs and total elapsed time

  **STATE_SLEEP changes (per FW-04 -- atomic trap guard):**

  Replace the current STATE_SLEEP block with an atomic check before entering deep sleep. The race condition: between configuring ext0 wakeup and calling `esp_deep_sleep_start()`, a trap could fire. ext0 wakeup only triggers DURING deep sleep, not before entering it -- the interrupt would be missed until the next scheduled wake (potentially hours).

  Fix: After configuring wake sources and powering down peripherals but immediately before `esp_deep_sleep_start()`, disable CPU interrupts, read the trap trigger pin directly, and if HIGH (trap has fired), abort sleep and re-enter the state machine.

  New implementation:

  ```cpp
  case STATE_SLEEP:
  {
    // Save config (in case commands updated it)
    storageSaveConfig(cfg);

    // Calculate next wake time
    uint64_t sleepUs = calculateNextSleep(cfg, rtc);
    Serial.printf("[SLEEP] Going to sleep for %llus\n", sleepUs / 1000000ULL);

    // Configure wake sources
    esp_sleep_enable_ext0_wakeup((gpio_num_t)TRAP_TRIGGER_PIN, 1); // HIGH = trap fired
    esp_sleep_enable_timer_wakeup(sleepUs);

    // Power down peripherals for deep sleep current savings
    gpsPowerOff();
    ledSet(LED_ALL, OFF);
    delay(100);

    // Atomic trap check before sleep entry (FW-04 fix)
    // Disable interrupts, read pin, decide whether to sleep or re-process.
    // ext0 wakeup only triggers DURING deep sleep -- if the trap fired between
    // configuring ext0 and entering sleep, it would be missed until next
    // scheduled wake (potentially hours).
    portDISABLE_INTERRUPTS();
    bool trapPending = (digitalRead(TRAP_TRIGGER_PIN) == HIGH);
    if (trapPending)
    {
      portENABLE_INTERRUPTS();
      Serial.println("[SLEEP] Trap trigger detected before sleep -- re-processing");
      wakeReason = WAKE_TRAP;
      state = STATE_WAKE_ASSESS;
      break;
    }
    portENABLE_INTERRUPTS();

    esp_deep_sleep_start();
    break; // Never reached
  }
  ```

  Key behaviors to ensure:
  - `portDISABLE_INTERRUPTS()` / `portENABLE_INTERRUPTS()` are FreeRTOS macros available on ESP32. They disable/enable CPU interrupts to create an atomic read section.
  - `digitalRead(TRAP_TRIGGER_PIN)` reads the physical pin state. Reed switch NC (normally closed) -- when trap fires, switch opens, pin goes HIGH.
  - If trap is pending: re-enable interrupts, set `wakeReason = WAKE_TRAP`, set `state = STATE_WAKE_ASSESS`, break. The next `loop()` iteration processes the trap event normally through the full state machine (WAKE_ASSESS -> COMPOSE_MSG -> GPS_CHECK -> MODEM_INIT -> SEND_MSG -> CMD_LISTEN -> MODEM_OFF -> SLEEP).
  - If no trap pending: re-enable interrupts and immediately call `esp_deep_sleep_start()`. The window between re-enable and deep_sleep_start is microseconds, and ext0 is already configured for any trigger during actual sleep.
  - GPS and modem are already powered off before this check. Re-entering WAKE_ASSESS will go through MODEM_INIT again -- this is correct since modem was powered off in STATE_MODEM_OFF.
  - The `break` after `esp_deep_sleep_start()` is never reached (CPU halts) but satisfies the switch statement syntax.
