    Structure the document as follows:

    ## Purpose
    Brief paragraph: This file is the authoritative inventory of every Supabase surface that downstream apps depend on. It is the companion to scripts/db-check.cjs, which enforces these rules statically. Before writing any migration that touches these surfaces, consult this document.

    ## Naming Convention
    One paragraph explaining the {namespace}_{NNN}_{description}.sql convention and the four namespace tokens: portal_, wildtrack_, fire_, trap_. Note that Trap Monitor has no migrations directory (schema applied via dashboard — this is a known governance gap).

    ## Portal Schema — Protected Surfaces
    Table with columns: Surface | Type | Signature / Key Columns | Consumers | Breakage if Changed | db-check ID

    Include ALL surfaces from the research inventory:
    1. portal.app_access — Table — id, user_id, app_id, role, granted_by, granted_at — WildTrack, Fire, Trap Monitor — Access gating breaks for all three apps — drop_portal_app_access / drop_column_app_access
    2. portal.profiles — Table — id, display_name, email, full_name, avatar_url, organisation, created_at, updated_at — WildTrack (display_name, email), all apps (id FK) — WildTrack member lookups break — drop_portal_profiles / drop_column_profiles
    3. portal.apps — Table — id, name, description, url, icon, created_at — All apps (registry) — App registry queries fail — drop_portal_apps
    4. portal.check_app_access(target_app_id text) — RPC function — Returns table(has_access boolean, user_role text) — WildTrack, Fire, Trap Monitor — All three apps' access gating fails immediately — drop_check_app_access / signature_change_check_app_access
    5. portal.is_admin() — Function — Returns boolean — portal admin UI — Admin actions break (portal only) — (no db-check; lower risk)
    6. portal.handle_new_user() — Trigger function — Inserts into portal.profiles on auth.users INSERT — All apps (new user signups) — Profile not created on signup across all apps — (no separate check; covered by drop_on_auth_user_created)
    7. on_auth_user_created — Trigger — AFTER INSERT on auth.users — All apps — Signup profile creation silently broken — drop_on_auth_user_created

    ## Public Schema — Cross-App Dependencies
    Table with columns: Surface | Type | Owner | Consumers | Breakage if Changed

    Include:
    1. public.organisations — Table — WildTrack — Trap Monitor (units.org_id FK) — Trap Monitor org FK breaks on CASCADE changes
    2. public.org_members — Table — WildTrack — Trap Monitor (trap_can_* functions query it) — Trap Monitor access checks fail
    3. public.is_org_member(p_org_id uuid, p_user_id uuid) — Function — WildTrack — Trap Monitor — Trap Monitor RLS policies using this function break
    4. public.is_org_admin(p_org_id uuid, p_user_id uuid) — Function — WildTrack — Trap Monitor — Trap Monitor admin checks break
    5. public.can_org_edit(p_org_id uuid, p_user_id uuid) — Function — WildTrack — Trap Monitor — Trap Monitor edit permission checks break

    ## Naming Collision Risks
    Table covering:
    1. public.update_updated_at() — Both WildTrack and Fire App define this function. CREATE OR REPLACE in either app silently overwrites the other's version if logic differs.
    2. Enum names in public schema — Currently distinct, but must be guarded. WildTrack: org_role, org_type. Fire App: user_role.

    ## Migration Filename Tracking Note
    One paragraph: Portal migration files were renamed from 001_portal_app_access.sql and 002_admin_policies.sql to portal_001_app_access.sql and portal_002_admin_policies.sql as part of Phase 1 Migration Governance. The live Supabase database still records the original filenames in supabase_migrations.schema_migrations. This divergence is intentional. Do NOT run `supabase db push` or `supabase migration up` for these renamed files — they are already applied to the live DB. All future migrations must use the namespace-prefix convention from the start.

    ## Rules for Schema Changes
    Number list (verbatim from CLAUDE.md "Rules for Schema Changes" section — 7 rules about never DROP/ALTER outside portal schema, never modify on_auth_user_created, never change check_app_access signature, etc.)

    ## Cross-App Impact Checklist
    Checkbox list (verbatim from CLAUDE.md "Cross-App Impact Checklist" section — 4 items).
