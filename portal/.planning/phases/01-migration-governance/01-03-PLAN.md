       - portal.profiles: profiles_read_own (SELECT, auth.uid() = id)
       - portal.profiles: profiles_update_own (UPDATE, auth.uid() = id)
       - portal.profiles: profiles_insert_own (INSERT, auth.uid() = id)
       - portal.profiles: profiles_read_admin (SELECT, portal.is_admin())

    2. Known Issue: Duplicate Policies in Portal Migrations
       Callout block:
       > KNOWN ISSUE: portal_001_app_access.sql and portal_002_admin_policies.sql both define app_access_insert_admin and app_access_delete_admin policies. In Supabase, policies with the same name on the same table are additive (both apply). The 001 version uses a direct subquery on portal.app_access as the admin check — this creates a recursive policy evaluation. The 002 version replaces this with portal.is_admin() (SECURITY DEFINER function, avoids recursion). The 001 non-definer INSERT/DELETE policies should be dropped in a future migration (portal_003_cleanup_recursive_policies.sql). This is a MIGR-06 finding, not a blocking issue in this phase.

    3. WildTrack Public Schema RLS (summary)
       Paragraph: WildTrack RLS is enabled on all tables. Policies use is_org_member(), is_org_admin(), can_org_edit() helper functions. These functions are WildTrack-owned but consumed by Trap Monitor's RLS — see PROTECTED_SURFACES.md for cross-app risk.

    4. Fire App Public Schema RLS (summary)
       Paragraph: Fire App RLS is enabled on all tables. Policies use auth.uid() and user_project membership checks. Isolated to Fire App tables — no cross-app risk from policy changes.

    5. Trap Monitor RLS (summary)
       Paragraph: Trap Monitor uses trap_can_view_org(), trap_can_edit_org(), trap_can_admin_org() functions on public schema tables. These functions query public.org_members (WildTrack-owned). Schema and policies were applied via Supabase dashboard, not migration files — full policy list unverifiable from source. See docs/schema/trap-monitor.md governance gap note.

    6. Findings Summary
       Bullet list of findings from this audit:
       - FINDING-1 (LOW): Portal migrations 001 and 002 define duplicate INSERT/DELETE policies on portal.app_access. Both are applied. The 001 versions use a recursive subquery (now superseded by 002 security-definer version). Future migration needed to clean up.

       - FINDING-3 (MEDIUM): Trap Monitor schema and RLS policies cannot be audited from source (no migration files). Cannot confirm policy completeness or correctness without live DB introspection.
